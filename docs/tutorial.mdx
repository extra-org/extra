## What we're building

**Nimbus Cloud** — a fictional cloud hosting company — needs a support assistant that covers two very different domains: account/billing issues, and platform/infrastructure issues. Getting this wrong is exactly where hallucination creeps in: an agent that half-knows billing *and* half-knows infrastructure will guess at both. So we split it properly.

```text
                    main_router
                   /            \
          account_router      platform_router
           /        \            /         \
   billing_agent  security_agent   infra_agent   docs_agent
   (local tool)   (protected)      (MCP)         (MCP)
```

Two orchestrators, four agents, a mix of a local tool and MCP servers — this is a realistic shape for a real company, not a toy demo. We'll build the YAML section by section, explaining *why* each decision keeps agents accurate and scoped, then generate, implement, and run it for real.

---

## Step 1 — Start with identity

Every file starts here. Nothing else exists yet:

```yaml agents.yml
system:
  name: "Nimbus Cloud Support"
```

That's a valid — if useless — spec. `system.name` is the only thing every single spec requires. Everything below gets added to this same file, one block at a time.

---

## Step 2 — Set a default model

```yaml agents.yml
system:
  name: "Nimbus Cloud Support"

defaults:
  model:
    provider: anthropic
    name: claude-haiku-4-5
    temperature: 0.0
```

**Why Haiku, and why `temperature: 0.0`?** Most of this system is routing and factual lookups, not creative reasoning. A fast, cheap, deterministic model is the right default. Cranking temperature down further reduces the chance an agent improvises instead of sticking to what it actually knows — the first lever against hallucination, before we've even written an agent.

---

## Step 3 — Declare the tool billing needs

```yaml agents.yml
tools:
  get_invoice_status:
    description: "Get the current invoice status and amount due for a customer account by account ID."
```

We declare this *before* any agent, because an agent will reference it by id in a moment — Extra needs `get_invoice_status` to already exist in this list, or the reference fails validation. Declaring capabilities before wiring them up is the pattern for the rest of the file: tools, then MCPs, then resolvers, *then* the agents that use them.

---

## Step 4 — Declare the MCP servers platform support needs

```yaml agents.yml
mcps:
  status_mcp:
    url: "https://mcp.nimbuscloud.internal/status"
  docs_mcp:
    url: "https://mcp.nimbuscloud.internal/docs"
```

<Note>
  These URLs are illustrative — they point at Nimbus Cloud's own internal servers, which don't exist outside this example. In your real deployment, this is where you'd put your company's actual internal MCP endpoints — an incident-status API, an internal docs search index, anything already running as an MCP server. The point of this step is that connecting one is exactly this simple: a name and a URL. No client code, no SDK.
</Note>

---

## Step 5 — Build the orchestrators, one at a time

Start with the two *sub*-routers — the ones closer to the actual work:

```yaml agents.yml
orchestrators:
  account_router:
    description: "Routes billing and account-security questions."
    prompts:
      orchestrator: "prompts/account_router/orchestrator.md"

  platform_router:
    description: "Routes infrastructure status and technical documentation questions."
    prompts:
      orchestrator: "prompts/platform_router/orchestrator.md"
```

**Why split into two routers instead of one flat list of four agents?** A single router choosing between "billing", "security", "infra status", and "docs" has to hold four unrelated domains in its head at once, using one prompt. Splitting into `account_router` and `platform_router` means each router only ever has to make an easy, narrow decision — and each one's prompt only needs to describe *its own* two children, not all four. Smaller decision surface, less room for the router to guess wrong.

Now add the top-level router that sits above both:

```yaml agents.yml
orchestrators:
  main_router:
    description: "Routes the user to account/billing support or platform/infrastructure support."
    prompts:
      orchestrator: "prompts/main_router/orchestrator.md"

  account_router:
    description: "Routes billing and account-security questions."
    prompts:
      orchestrator: "prompts/account_router/orchestrator.md"

  platform_router:
    description: "Routes infrastructure status and technical documentation questions."
    prompts:
      orchestrator: "prompts/platform_router/orchestrator.md"
```

`main_router` doesn't need to know that `billing_agent` or `docs_agent` exist at all — it only ever sees `account_router` and `platform_router`, via their `description` fields. This is the same accuracy principle one level up: each orchestrator's world is exactly as big as it needs to be, no bigger.

---

## Step 6 — Add the agents

```yaml agents.yml
agents:
  billing_agent:
    description: "Handles invoice status, billing amounts, and payment questions."
    prompts:
      system: "prompts/billing/system.md"
    tools: [get_invoice_status]

  security_agent:
    description: "Handles account security: password resets, 2FA, and suspicious login reports."
    prompts:
      system: "prompts/security/system.md"
    protected: true

  infra_agent:
    description: "Reports on current infrastructure status and ongoing incidents."
    prompts:
      system: "prompts/infra/system.md"
    mcps: [status_mcp]

  docs_agent:
    description: "Answers technical questions using Nimbus Cloud's API and product documentation."
    prompts:
      system: "prompts/docs/system.md"
    mcps: [docs_mcp]
```

Four agents, four completely separate scopes:

- `billing_agent` gets exactly one tool: `get_invoice_status`. It has no MCP access and no security tools — it structurally *cannot* touch account security, even if a user tries to steer it there.
- `security_agent` is `protected: true` — more on this in [Step 9](#step-9-protect-the-sensitive-agent).
- `infra_agent` and `docs_agent` each get exactly one MCP server. `infra_agent` can't accidentally answer from documentation, and `docs_agent` can't accidentally report live incident status — they're reading from different systems entirely.

This is the actual mechanism behind "no hallucination bleed between domains": it's not a prompt instruction asking the model to behave — it's that each agent is *structurally* wired to only one slice of tools and data. There's nothing else for it to reach for.

---

## Step 7 — Write the prompt files

The YAML above references six prompt files that don't exist on disk yet. `agentctl generate` (next step) only scaffolds *plugin code* — tools, resolvers, hooks. It never writes prompt files for you, and `validate` will fail if a referenced prompt file is missing. So we write these first.

```markdown prompts/main_router/orchestrator.md
You are the top-level router for Nimbus Cloud Support.
Route account and billing questions to account_router.
Route infrastructure status and technical documentation questions to platform_router.
```

```markdown prompts/account_router/orchestrator.md
You route between billing and account security.
Route invoice, payment, and billing amount questions to billing_agent.
Route password, 2FA, and suspicious login questions to security_agent.
```

```markdown prompts/platform_router/orchestrator.md
You route between infrastructure status and documentation.
Route questions about outages, incidents, or current system status to infra_agent.
Route questions about how to use the API or product features to docs_agent.
```

```markdown prompts/billing/system.md
You are a billing specialist for Nimbus Cloud.
Use the get_invoice_status tool to check real invoice data before answering.
Never guess an amount or a due date — always look it up.
```

```markdown prompts/security/system.md
You are an account security specialist for Nimbus Cloud.
You only assist customers whose identity has already been verified.
Be precise about what security actions were taken.
```

```markdown prompts/infra/system.md
You are an infrastructure status specialist for Nimbus Cloud.
Report only what the status tool returns — do not speculate about causes or resolution times you weren't given.
```

```markdown prompts/docs/system.md
You are a documentation specialist for Nimbus Cloud's API and product.
Answer using the documentation search tool. If it doesn't have an answer, say so instead of guessing.
```

Notice the repeated pattern: every prompt tells the agent to defer to its tool's real data rather than "reason" its way to an answer. That instruction only works *because* the agent has nothing else available to reach for — the prompt and the structure reinforce each other.

---

## Step 8 — Generate the plugin stub

Only `billing_agent`'s tool needs real code — `infra_agent` and `docs_agent` get their capabilities from MCP servers, which need no local implementation at all.

```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  ghcr.io/asaf-prog/extra:latest \
  generate --config agents.yml
```

This creates:

```text
plugins/
  __init__.py
  plugins.toml
  tools/
    __init__.py
    get_invoice_status.py
```

`plugins/tools/get_invoice_status.py`, freshly generated:

```python plugins/tools/get_invoice_status.py
def get_invoice_status(account_id: str) -> dict:
    """Get the current invoice status and amount due for a customer account by account ID."""
    raise NotImplementedError
```

`raise NotImplementedError` is a deliberate placeholder, not an error in the tool — it won't stop `generate` or `validate` from succeeding. If the model tries to call this before you implement it, you get a loud, clear failure instead of silent wrong behavior. `plugins/plugins.toml` is the manifest `generate` wrote mapping `get_invoice_status` to this file — **you never hand-edit this file**; it's regenerated as your tools change. That's a different thing entirely from a Python dependency file, which is next.

---

## Step 9 — Protect the sensitive agent {#step-9-protect-the-sensitive-agent}

`security_agent` is marked `protected: true` in Step 6. Any spec with a protected node requires `plugins/access.py`:

```python plugins/access.py
class AccessResolver:
    def can_access(self, ctx, node_id: str) -> bool:
        if node_id == "security_agent":
            return ctx.headers.get("X-Customer-Verified") == "true"
        return True
```

Until a caller's request carries that header, `security_agent` is invisible to `account_router` entirely — not refused, not filtered after the fact, just never offered as an option. That's a stronger guarantee than anything you could put in a prompt.

## Step 10 — Implement the tool

Real companies don't expose their billing database as a public API, so we simulate one with an in-memory lookup — in production, swap this for your real database or internal billing service call:

```python plugins/tools/get_invoice_status.py
_INVOICES = {
    "acct-1001": {"status": "paid", "amount_due": 0.00, "due_date": None},
    "acct-1002": {"status": "overdue", "amount_due": 149.00, "due_date": "2026-06-15"},
}

def get_invoice_status(account_id: str) -> dict:
    """Get the current invoice status and amount due for a customer account by account ID."""
    invoice = _INVOICES.get(account_id)
    if invoice is None:
        return {"error": f"No account found with ID '{account_id}'."}
    return {"account_id": account_id, **invoice}
```

No new dependency needed here — this tool uses nothing beyond the standard library. If yours does (an HTTP client, a database driver), that goes in `requirements.txt` at the project root — a plain pip file, not TOML, and a different mechanism entirely from `plugins.toml`.

---

## Step 11 — Validate

```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  ghcr.io/asaf-prog/extra:latest \
  validate agents.yml
```

Fully offline. It checks the YAML shape, that every tool/MCP/resolver id referenced by an agent is actually declared, that all six prompt files exist on disk, and that `plugins/access.py` is importable since a protected node exists. Exits non-zero with a specific error location if anything's wrong.

---

## Step 12 — Run it

```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  -e ANTHROPIC_API_KEY=sk-... \
  ghcr.io/asaf-prog/extra:latest \
  run --config agents.yml --message "What's the status of invoice for account acct-1002?"
```

The path: `main_router` → `account_router` → `billing_agent` → `get_invoice_status("acct-1002")` → a real, looked-up answer: overdue, $149.00, due 2026-06-15.

```bash
docker run --rm \
  -v $(pwd):/workspace \
  -w /workspace \
  -e ANTHROPIC_API_KEY=sk-... \
  ghcr.io/asaf-prog/extra:latest \
  run --config agents.yml --message "Is there an ongoing outage right now?"
```

This one routes `main_router` → `platform_router` → `infra_agent`. Since `status_mcp` points at an illustrative URL that doesn't really exist, the engine logs a warning and connects with no tools bound for that agent — it'll answer from the prompt alone, without a real status lookup. Point `status_mcp`'s `url` at your actual internal server and this same call starts returning real incident data, with no other change to the spec.

## What you just built

- A two-level routing hierarchy where every orchestrator's decision space is as narrow as possible
- Four agents with completely non-overlapping tool/MCP access — the actual mechanism behind avoiding cross-domain hallucination
- One protected node, gated by a real access plugin instead of prompt wording
- A real, working local tool, plus two MCP declarations ready to point at real internal servers
