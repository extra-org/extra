## Overview

A spec file has two conceptual halves:

- **Declarations** — what exists: agents, orchestrators, tools, MCP servers, resolvers, hooks.
- **Graph** — how they connect at runtime. Indentation is the topology.

The engine validates and compiles the YAML before any request runs. It never executes YAML content directly.

```yaml
# Everything you declare...
system: ...
defaults: ...
mcps: ...
tools: ...
resolvers: ...
orchestrators: ...
agents: ...
hooks: ...

# ...and how it connects
graph:
  root_orchestrator:
    child_agent_a:
    child_agent_b:
```

## Running example

All examples on this page build toward a customer support bot for an online store.
It routes users to the right department, each with its own tools and data sources.

<Frame>
```
User message
     │
     ▼
  router          ← orchestrator: decides where to send the request
  ├── orders_agent     ← handles order status, tracking
  ├── returns_agent    ← handles refunds and returns
  └── catalog_agent    ← handles product questions
```
</Frame>

## `system`

Metadata about your system. Currently used for display and tracing.

```yaml
system:
  name: "Acme Support Bot"
```

| Field | Required | Description |
|-------|----------|-------------|
| `name` | yes | Human-readable name shown in logs and traces. |

---

## `defaults`

Default model for every agent and orchestrator. Can be overridden per node.

```yaml
defaults:
  model:
    provider: anthropic
    name: claude-sonnet-4-6
    temperature: 0.3
```

<Tip>
  Set a cheap/fast model as the default (e.g. Haiku) and override to a stronger one only for agents that need it. Orchestrators doing only routing work well with a fast model.
</Tip>

### Supported providers

<CodeGroup>
```yaml Anthropic
model:
  provider: anthropic
  name: claude-sonnet-4-6
  temperature: 0.3
```

```yaml Amazon Bedrock
model:
  provider: bedrock
  name: anthropic.claude-3-5-haiku-20241022-v1:0
  region: us-east-1
  temperature: 0.0
```
</CodeGroup>

For Bedrock, `region` can be omitted if `AWS_REGION` or `AWS_DEFAULT_REGION` is set. AWS credentials follow the standard chain (`AWS_PROFILE`, `AWS_ACCESS_KEY_ID`, IAM roles, etc.).

<Warning>
  Never put API keys or secrets in YAML. Use environment variables — the engine reads them at runtime.
</Warning>

## `orchestrators`

Orchestrators are **routers**. They read the user's message, look at their children's descriptions, and decide which child to invoke. They do not call tools or MCP servers directly — their children are their capabilities.

```yaml
orchestrators:
  router:
    name: "Support Router"
    description: "Routes customer requests to the right department."
    model:
      provider: anthropic
      name: claude-haiku-4-5    # routing only — fast model is fine
      temperature: 0.0
    prompts:
      orchestrator: "prompts/router/orchestrator.md"
      system: "prompts/router/system.md"   # optional
```

| Field | Required | Description |
|-------|----------|-------------|
| `name` | no | Display name. Defaults to the key. |
| `description` | no | Used by parent orchestrators to decide when to route here. |
| `model` | no | Overrides `defaults.model`. Full replacement — specify all fields. |
| `prompts.orchestrator` | **yes** | Instructions for how to route among children. |
| `prompts.system` | no | Additional system context. |
| `prompts.user` | no | User-turn context injection. |

### The orchestrator prompt

The `orchestrator` prompt is what tells the LLM how to choose among its children. It sees the children's `description` fields. Example `prompts/router/orchestrator.md`:

```markdown
You are a customer support router for an online store.
Route the user to the most relevant department based on their message.
If unclear, ask one clarifying question before routing.
```

## `agents`

Agents are **executors**. They receive the user's message (after routing), run an LLM with their prompt, and may call tools or MCP servers to fulfill the request.

```yaml
agents:
  orders_agent:
    name: "Orders"
    description: "Handles order status, tracking, and delivery questions."
    model:
      provider: anthropic
      name: claude-sonnet-4-6
    prompts:
      system: "prompts/orders/system.md"
      user: "prompts/orders/user.md"       # optional
    resolvers: [current_date, customer_name]
    tools: [get_order_status, cancel_order]
    mcps: [orders_mcp]

  returns_agent:
    name: "Returns"
    description: "Handles return requests, refund status, and return policy questions."
    prompts:
      system: "prompts/returns/system.md"
    tools: [create_return_request]
    mcps: [orders_mcp]

  catalog_agent:
    name: "Catalog"
    description: "Answers questions about products, availability, and specifications."
    prompts:
      system: "prompts/catalog/system.md"
    mcps: [catalog_mcp]
```

| Field | Required | Description |
|-------|----------|-------------|
| `name` | no | Display name. |
| `description` | **yes (if has a parent orchestrator)** | How the orchestrator knows when to route here. |
| `model` | no | Overrides `defaults.model`. |
| `prompts.system` | no | System prompt file. |
| `prompts.user` | no | User-turn context injection. |
| `resolvers` | no | List of resolver ids to run before this agent. |
| `tools` | no | List of tool ids this agent can call. |
| `mcps` | no | List of MCP server ids this agent has access to. |
| `protected` | no | `true` to enable access control for this node. See [Access Control](#access-control). |

### The system prompt

The `system` prompt defines the agent's behavior and constraints. Example `prompts/orders/system.md`:

```markdown
You are an orders specialist for Acme Store.
Today's date is {{current_date}}.
You are helping {{customer_name}}.

You can:
- Check order status and tracking
- Cancel orders that haven't shipped yet
- Escalate to a human agent for complex issues

You cannot process refunds — direct those questions to the returns team.
```

Variables like `{{current_date}}` are filled by resolvers before the agent runs.

## `tools`

Tools are Python functions exposed to the LLM. The agent decides at runtime whether to call them based on the conversation.

```yaml
tools:
  get_order_status:
    description: "Get the current status and tracking info for an order by order ID."

  cancel_order:
    description: "Cancel an order that hasn't shipped. Requires order ID and reason."

  create_return_request:
    description: "Open a return request for a delivered order. Requires order ID and reason."
```

| Field | Required | Description |
|-------|----------|-------------|
| `description` | yes | What the LLM sees when deciding whether to call this tool. Write it clearly — it directly affects tool selection. |

The actual implementation lives in `plugins/tools/<tool_name>.py`, generated by `agentctl generate`. You fill in the business logic.

<Tip>
  Write tool descriptions as if explaining to a smart assistant. "Get order status by order ID" is better than "order_status_getter". The description is the LLM's only signal.
</Tip>

## `mcps`

MCP servers expose remote tools discovered at build time. The engine creates one client per server and binds its tools to agents that declare it.

```yaml
mcps:
  orders_mcp:
    url: "https://api.acme.com/mcp"

  catalog_mcp:
    url: "https://catalog.acme.com/mcp"
    tool_tags: ["public"]        # only discover tools tagged "public"
```

| Field | Required | Description |
|-------|----------|-------------|
| `url` | yes | Streamable HTTP MCP endpoint. |
| `tool_tags` | no | Only discover tools with this tag. Sent as `X-MCP-Tool-Tag` header. |
| `tool_tag_transport` | no | Override how tags are sent. See below. |

### Tool tags

If your MCP server groups its tools by tag, you can limit which tools each agent sees:

```yaml
mcps:
  crm_mcp:
    url: "https://crm.acme.com/mcp"
    tool_tags: ["read-only"]     # orders_agent only gets read tools

  crm_mcp_admin:
    url: "https://crm.acme.com/mcp"
    tool_tags: ["admin"]         # admin_agent gets write tools
```

For servers that use a query param instead of a header:

```yaml
mcps:
  legacy_api:
    url: "https://legacy.acme.com/mcp"
    tool_tags: ["v2"]
    tool_tag_transport:
      type: query_param
      param_name: "tag"
```

## `resolvers`

Resolvers fill `{{variables}}` in prompt files **before** the agent runs. They are deterministic — the LLM never sees them, never chooses them, and they cost no tokens.

```yaml
resolvers:
  current_date:
    scope: shared      # generated once, available to all agents

  customer_name:
    scope: shared

  loyalty_tier:
    scope: agent       # generated per-agent, only for agents that declare it
```

| `scope` | Description |
|---------|-------------|
| `shared` | Runs once on `SharedResolver`. All agents inherit it. Use for things like current date, user identity. |
| `agent` | Runs only for the agents that declare it. Use for agent-specific context. |

Implementation is generated by `agentctl generate` into `plugins/resolvers/`. Example:

```python
class SharedResolver:
    def current_date(self, ctx) -> str:
        return datetime.now().strftime("%Y-%m-%d")

    def customer_name(self, ctx) -> str:
        return ctx.headers.get("X-Customer-Name", "Customer")
```

### Resolvers vs. tools

Both point at Python code, but they're different:

| | Resolver | Tool |
|---|---|---|
| When it runs | Before the LLM | During LLM execution |
| Chosen by | Engine (always) | LLM (when it decides to) |
| Visible to LLM | No | Yes (name + description) |
| Token cost | None | Yes |
| Purpose | Fill prompt variables | Perform actions |

## `hooks`

Hooks are trusted lifecycle code — auth, policy, audit — that run automatically and are **never exposed to the LLM**.

```yaml
hooks:
  before_mcp_request:
    - plugin: mcp_auth
      method: before_mcp_request
      config:
        credential_env: INTERNAL_MCP_SECRET
```

The most common hook is `before_mcp_request`: it injects `Authorization` headers before each MCP call, so private MCP servers work without putting tokens in prompts or logs.

See [Runtime Hooks](/docs/runtime-hooks) for all 10 lifecycle points and examples.

## `graph`

The graph declares runtime topology. Indentation = parent/child relationship. The single top-level key is the root — the first node every request hits.

```yaml
graph:
  router:
    orders_agent:
    returns_agent:
    catalog_agent:
```

More complex example — nested orchestrators:

```yaml
graph:
  main_router:
    store_router:          # sub-orchestrator for store questions
      orders_agent:
      returns_agent:
      catalog_agent:
    account_agent:         # direct agent for account management
    vip_agent:             # protected — only accessible to VIP customers
```

Rules:
- Exactly one root.
- Every key must be declared in `orchestrators` or `agents`.
- Orchestrators can have children; agents are normally leaves.
- No cycles.

## `plugins`

Controls how the engine finds your plugin code when running from different directories.

```yaml
plugins:
  import_roots: ["."]    # make the project root importable
```

This lets you use package-style references in hooks (`company.plugins.auth:AuthHook`) without worrying about the working directory. Resolved relative to the spec file.

## Access control

Mark any node `protected: true` to gate it behind your access plugin:

```yaml
agents:
  vip_agent:
    description: "Exclusive offers and services for VIP members."
    protected: true
    prompts:
      system: "prompts/vip/system.md"
```

When any node is `protected: true`, the engine expects `plugins/access.py`:

```python
class AccessResolver:
    def can_access(self, ctx, node_id: str) -> bool:
        # ctx has request headers and metadata
        return ctx.headers.get("X-Customer-Tier") == "vip"
```

Protected nodes that fail the check are hidden from the router entirely — the LLM never sees them.

## Complete example

An e-commerce support bot with two tiers of customers:

```yaml
system:
  name: "Acme Customer Support"

defaults:
  model:
    provider: anthropic
    name: claude-haiku-4-5
    temperature: 0.0

mcps:
  orders_mcp:
    url: "https://api.acme.com/mcp"
    tool_tags: ["support"]
  catalog_mcp:
    url: "https://catalog.acme.com/mcp"

tools:
  create_return:
    description: "Open a return request for a delivered order by order ID and reason."

resolvers:
  current_date:
    scope: shared
  customer_name:
    scope: shared

orchestrators:
  router:
    description: "Routes customer requests to the right department."
    model:
      provider: anthropic
      name: claude-haiku-4-5
      temperature: 0.0
    prompts:
      orchestrator: "prompts/router/orchestrator.md"

agents:
  orders_agent:
    description: "Order status, tracking, and delivery questions."
    prompts:
      system: "prompts/orders/system.md"
    resolvers: [current_date, customer_name]
    mcps: [orders_mcp]

  returns_agent:
    description: "Return requests and refund questions."
    prompts:
      system: "prompts/returns/system.md"
    resolvers: [current_date, customer_name]
    tools: [create_return]
    mcps: [orders_mcp]

  catalog_agent:
    description: "Product questions, availability, and specifications."
    prompts:
      system: "prompts/catalog/system.md"
    mcps: [catalog_mcp]

  vip_agent:
    description: "Exclusive deals and priority support for VIP members."
    protected: true
    prompts:
      system: "prompts/vip/system.md"
    resolvers: [customer_name]
    mcps: [orders_mcp]

hooks:
  before_mcp_request:
    - plugin: mcp_auth
      method: before_mcp_request
      config:
        credential_env: ACME_MCP_SECRET

plugins:
  import_roots: ["."]

graph:
  router:
    orders_agent:
    returns_agent:
    catalog_agent:
    vip_agent:
```
